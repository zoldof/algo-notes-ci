# コーディング
- 詰まったら順番にコード内容をAIに投げて質問する
- つまずいたらコミット履歴を見る

# マークダウン
- テスト用はmarkedでの反映
- ビルド用はkramdownでの反映

＊使い分けも可能だがビルド用閲覧で統一するならkramdown記法に沿って記述する
＊初期のkramdown導入は事前ビルド構成になるため設計意図と異なる

# ブランチ構成
- ［test］test.html→［main］←［main］index.html
  - こちらはパス構造ではないので気を付ける
- 意図はそれぞれ［テスト，ビルド，自分用表示］
- テスト表示だとフロントマターは残存する
  - デフォルトビルド後は、一旦テスト表示ができなくなる
- ［test］でテスト［main］でビルドの流れを把握しつつ、各ファイルの更新と開発段階に合わせて柔軟にデフォルトブランチを変更する手筈になっている

# プルマージの注意点
- 削除履歴も反映されるので必要なものは残してマージする

# ビルド設定
- 静的化は［test］で常時動作でも負担は少ない
  - ビルドを全面停止したい場合はstatic.ymlのpushをコメントアウトする
- 動的化は［main］で停止したい場合はビルド構成をアクションに設定する

# ページへの反映
- あくまでもビルド対象のブランチおよびディレクトリが参照される
  - つまりは静的と動的で共有使用するスクリプトがあった場合、最後にビルドされた方式が優先される
- ブランチが異なっても、正体も階層構造も一緒でページの参照先としても同じ
  - デフォルトビルドの対象ブランチが変更されていても同様という話
  - デフォルトブランチが変更されていても同様
- テスト対応として前回正常動作構成に設定しても直るまで時間差出る場合あり
