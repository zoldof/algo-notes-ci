# コーディング
- 詰まったら順番にコード内容をAIに投げて質問する

# マークダウン
- テスト用はmarkedでの反映
- ビルド用はkramdownでの反映

＊使い分けも可能だがビルド用閲覧で統一するならkramdown記法に沿って記述する
＊初期のkramdown導入は事前ビルド構成になるため設計意図と異なる

# ブランチ構成
- test/test.html→main←main/index.html
  - ここでの「/」はパス構造ではないので気を付ける
- 意図はそれぞれ［テスト，ビルド，自分用表示］
- テスト表示だとフロントマターは残存する

# プルマージの注意点
- 削除履歴も反映されるので必要なものは残してマージする

# ビルド設定
- 静的化はtestで無視できる範囲
  - ビルドを全面停止したい場合はconfigの指定ブランチ名を適当な名前に変更する
- 動的化はmainで停止したい場合はビルド構成をアクションに設定しておく

# ページへの反映
- 同じファイルの場合はmainのものが優先反映される
  - デフォルトブランチが変更されていても同様
  - デフォルトビルドの対象ブランチが変更されていても同様
  - したがって、テスト作業時は該当htmlをmainから削除しておく必要あり
- テスト対応として前回正常動作構成に設定しても直るまで時間差出る場合あり

- モジュールスクリプトのキャッシュは自動的に最適化される場合が多い
