## HTML[^](#top)
### 一般的なページ構造
上部にナビゲーションや現在位置の把握を担うヘッダー、中央に主要な情報や機能を提供するメイン領域、そして下部に補助情報を集めたフッターという三層構造で整理される。この中でフッターは、「今すぐ使うわけではないが、存在しないと困る情報」を扱う役割を持つ。具体的には、利用規約やプライバシーポリシー、問い合わせ窓口、サイト全体のリンクといった要素が該当する。最下部のコントロールバーは別の扱いになる。

### !DOCTYPE htmlの必要性
理論的には省略可能であるものの、挙動の安定性と再現性を確保するという観点から、実務上は常に記述すべき要素と位置づけられる。この宣言が存在する場合、ブラウザは文書を標準モードで処理する。標準モードでは、HTMLやCSSの仕様に基づいた一貫性のあるレイアウト計算が行われるため、開発者が意図した通りの表示が再現されやすい。一方、この宣言が省略された場合や不適切な形式で記述された場合、ブラウザは互換モードに移行する可能性がある。互換モードでは、過去のブラウザとの互換性を保つために古い仕様や独自挙動が再現されるため、同じコードであってもレイアウトやスタイルの適用結果が変化することがある。

## JavaScript[^](#top)
### クォーテーション
シングルとダブルは、外側と内側で入れ子にできる、同種ならエスケープが必要な点についても同じ。それらとは別にバッククォートは、文字列展開ができ、複数行文字列がそのまま書け、タグ付けテンプレートが使え、内部ではエスケープなしでシングルとダブルが使える。

### DOMContentLoaded
DOM操作は div id="content" やヘッダー要素などが DOM に存在して初めて意味を持ちます。jsが body の末尾に置かれている場合、DOMContentLoadedを使ったところで、実際には DOM がすでに構築された状態で実行されますが、将来的にjs位置をヘッド側へ移したり、外部ファイル化したりする可能性 を考慮すると、DOMContentLoaded にハンドラを登録しておく方が安全です。イベントが発火すれば、ページの構造が変わっても要素取得が失敗するリスクを回避でき、コードの保守性が向上します。

### モジュールの必要性
js内でimportやexportを使う場合に必要である。スコープ衝突やグローバル汚染を防ぎたい場合や遅延ロードの目的（DOMContentLoadedにより異なるjs間の変数や関数の呼び出しを明示的に順序化する場合）で使う。importを含むjs内でのfetchであれば必要になる。当然ながらjsをモジュール化した場合は外部のインラインハンドラは機能しなくなる。

### モジュール引数の必要性
html要素が設定されているならjs内部で呼び出し可能。ただし、DOM操作後に動かしたい場合や別の呼び出し機構が存在する場合には、引数を付けることで汎用性や保守性を担保できる。

### モジュール内のプライベート関数
モジュールが読み込まれればトップレベルで addEventListener が評価され、即座にリスナーが登録される。リスナーは対象オブジェクト（document、window、DOM要素）に対して動作し、export の有無は無関係。したがって、関数にまとめなくてもモジュール内部のプライベート関数を自由に呼び出せる。リスナーに対してモジュール全体にかかる変数と関数は区別して扱う。特にリスナー内部で変数を定義すると、実行されるまで存在せずロード時の実行ではエラーの原因になる。

### スコープ
ページ全体に対して1つのグローバルオブジェクトのwindowが与えられる。使用する場合は名前空間を用いる方が安全である。js自体はグローバルでhtml全体を通して定義された変数を使える。ただし、type="module"で読み込む場合は、js内の変数や関数はその内部でのみ扱うことができ、他のjsで呼び出せない。呼び出して使いたい場合はimport/exportが必要になる。この機能を使うためには同じくtype="module"である必要があるので、付け忘れに注意する。、クロージャは「内部関数 + その捕捉した環境」であり、カプセル化を可能とする。クラスはメソッド・フィールドを集合させた設計図で、# を使うと プライベートフィールド・メソッド が実装できる。関数だけでもローカル変数やクロージャで同様の隠蔽ができ、インスタンス化は不要。どちらを使っても 外部から利用する際は必ずモジュールで export し、import で取り込む 必要がある。したがって 「クラスか関数か」よりも「モジュールで公開するかしないか」 が、外部からのアクセス制御の決定要因になる。

### 要素の属性data
ある要素についてdata-csv="data.csv"のような書き方で属性を追加できる。これはデータが element.dataset に自動的にマッピングされる仕組み。

### 正規表現への対応
RegExpを使えばエスケープなしで文字列検索に対応できる。動的に生成した正規表現自体を変数に格納して使える。

### 配列の削除
popで最後尾から削除したり、spliceをつかう。spliceでは削除方向を±で指定、開始位置を添え字引数で指定でき、別引数で何文字取り出すかを決める。

## CSS[^](#top)
### IDとクラス
個別の要素に設定を反映する場合はID、複数の要素の場合はクラスをつかう。特に装飾においてクラスをつかう。

### ビューポートの扱い
bodyの高さを指定する場合は、htmlの高さに紐付けして書く方が良い。htmlの高さを指定しない場合、暗黙的にheadの高さが決まるため、bodyの位置がずれるから。

### flexの扱い
bodyすなわち親側でflexに設定すると子要素の中央寄せが簡単にできる。さらに、子要素がfixedだった場合には、子要素の位置設定が優先されるので中央寄せしたい要素と区別して扱うことができる。

## パス指定について[^](#top)
### 前提
相対パスか絶対パスかの選択は，「リポジトリをまたぐかどうか」ではなく URL 空間がどう定義・制御されているかで決まる。したがって，議論の基準は常に「ディレクトリ」ではなく URL である。

### 相対パス
- ［./］：省略も可能：同じDF（ディレクトリにあるファイル名を指定する場合の書き方）
  ↑moduleのjsでimportを使用する場合はパッケージ認識となるため省略不可
  ↑import は最終的には常にURLを解決している仕組みであり、ブラウザはファイル構成やリポジトリ構造を一切知らない
- ［../］：上位階層のDF
- ［../../］：2つ上の階層のDF
  ↑相対パスの場合、呼び出し元（パスコードを記入するファイル）からの位置指定である

### ブラウザ解決規則
例えばindex.htmlからmoduleのmain.jsを呼び出しfetchを使用する場合、htmlから見た相対パスを記述する

### 相対パスの補足説明
相対パスは「今見ている URL を基準」に解決されるため，表示 URL が変わると 同じ HTML / JS でも読み込み先が変化してしまう。その結果，相対パスを使うと「どの URL で表示されたか」を常に意識する必要が生じ，思考負荷と事故の余地が増える。

### GitHub Pagesのパス構造
GitHub Pages などでは，import は GitHub リポジトリを見ているのではなく，公開された URL をそのまま取得している。URL とファイル配置が一致している静的環境では，相対パスは自然で直感的に使える。

### 絶対パス
- ［/］：リポジトリの置いてあるサイトルートDF
- ［<base>］：設定パスを基準として［/］と同じ働きをするが、特にモジュール内部でimportする場合など扱いが煩雑になりがちなので、出来るだけ使用を避ける
  ↑kramdownの脚注の参照位置がずれる
  ↑URL基準（リンクや画像の相対パス）を決めるだけで、カスタム属性の属性は触れない

### Jekyll使用時の絶対パス
- ［baseurl］：リポジトリ直下を既定値とし_config.ymlに記述してhtmlで設定可
  ↑独自ドメインを使用する場合［baseurlは空，PagesのURLは変更］になるので別の変数を設定する
  ↑フロントマターのあるファイルにのみ機能する

＊呼び出し元がhtmlでもjsでも基本は同じ書き方
＊ビルド設定によって公開ディレクトリを変更した場合、相対パス指定も同調してずれる

### 絶対パスの補足説明
ルーティングと asset 読み込みを分離するため，動的 HTML を使う環境では origin 基準（/ 始まり）のパスを使う設計が好まれる。これは「絶対パスが正しい」という意味ではなく，表示 URL に依存しない参照点を選んでいるにすぎない。フレームワークやバンドラーは，この分離を前提に asset の URL を一元管理し，開発者が表示 URL を意識しなくて済むようにしている。

### ルーティングのパス構造
ルーティングを持つシステムでは，ブラウザに表示される URL は必ずしも実在するファイルの場所を指していない。SPA や SSR では，/posts/123 や /user/me のような URL は，ファイルではなく ルーターの規則によって動的に生成された表示用パスである。この場合，HTML や JS が実際に配置されている場所（例：/assets/app.js）と，表示 URL の階層は無関係になる。

### バンドラーのパス構造
バンドラーを使う場合，開発時の import はファイルシステム（プロジェクトルート，ディレクトリ構成）を基準に解釈され，その結果がビルド時に URL ベースの参照へ変換される。# や @ などでプロジェクトルートを指す書き方は JavaScript やブラウザの仕様ではなく，バンドラーが独自に解釈しているだけである。バンドラーで「絶対パス風」に書いていても，それは URL の絶対パスではなく，ビルド前の便宜的な記法にすぎない。バンドラーを使わずにブラウザが直接 import を解釈する場合，使えるのは相対 URL，origin 基準の絶対 URL（/ 始まり），完全 URL のみである。このときの「絶対」は プロジェクトルートではなく origin 基準であり，/ は「配信元のルート」を意味する。

# マークダウン[^](#top)

# プルマージの注意点
- 削除履歴も反映されるので必要なものは残してマージする
- READMEはともかく、マージにはすべてのディレクトリとファイルの構成一致が必要である

# ビルド設定
- 静的化にあたるアクションは［test］で常時動作でも負担は少ない
  - ビルドを全面停止したい場合はstatic.ymlのpushをコメントアウトする
  - まとめて編集ならボタン操作で十分
- 動的化にあたるビルドデプロイを［main］で停止したい場合はビルド構成をアクションに設定する

# デプロイについて
- ファイルのアップロードによって初めて各ファイルの更新が反映される
  ↑Pagesであっても自前サーバーであっても
  ↑StaticでもJekyllでも

＊Staticのymlではビルド工程はないので「Staticビルド」との呼称は改める必要有

# ページへの反映
- あくまでもビルド方法による
- ブランチが異なっても、正体も階層構造も一緒でページの参照先としても同じ
  - デフォルトビルドの対象ブランチが変更されていても同様という話
  - デフォルトブランチが変更されていても同様
- テスト対応として前回正常動作構成に設定しても直るまで時間差出る場合あり

# キャッシュの管理[^](#top)
- モジュールスクリプトのキャッシュは自動的に最適化される場合が多い
- CDNのエッジキャッシュのTTLが長いことから、cdn_baseとrevisionの合わせ技でコミットSHAを元にファイル判別する仕組みとした
  ↑この仕組みを使うとCDNに展開しているファイルを更新してビルドした時点で、_layoutを介したファイルの全てが影響を受けて最新版に更新される
  ↑全コミットに対する関連ファイル全ビルドを避けるため、テスト実行に位置付けた
  ↑直接パージURLを叩けば本番更新できるが、リクエスト上限があるため頻繁な更新には向かない
  ↑パージによる更新については、反映まで数分かかる場合あり
