# コーディング  
- 詰まったら順番にコード内容をAIに投げて質問する

## HTML
### 一般的なページ構造
上部にナビゲーションや現在位置の把握を担うヘッダー、中央に主要な情報や機能を提供するメイン領域、そして下部に補助情報を集めたフッターという三層構造で整理される。この中でフッターは、「今すぐ使うわけではないが、存在しないと困る情報」を扱う役割を持つ。具体的には、利用規約やプライバシーポリシー、問い合わせ窓口、サイト全体のリンクといった要素が該当する。最下部のコントロールバーは別の扱いになる。

### ホワイトスペース正規化
HTMLのパーサはテキストノード内の改行、タブ、複数のスペースをすべて単一のスペース文字 (U+0020) に変換します。したがって、要素間に改行を書いても画面上にはスペース1つだけが表示され、改行としては見えません。

### headとbody
`<head>`：書類の表紙に書かれた「タイトル」や「作成者」「日付」など、内容そのものではなく「この文書は何か」を示す情報。head に置かれた script はデフォルトで HTML の解析が始まる前に実行 される（defer や async を付けない限り）。meta（例: description, robots, og:*）や title は SEO / SNS シェア に直接影響する。\
`<body>`：実際の本文――段落や図表、写真が並び、読者が読む部分。body 内の script は その位置までの要素が描画された後 に実行されることが多い。\
このように、HTML は 設定情報をまとめる`<head>`と 閲覧者が目にする`<body>`に明確に分けることで、ページの構造と動作を整理し、ブラウザや検索エンジンが正しく処理できるようにしています。

### !DOCTYPE htmlの必要性
理論的には省略可能であるものの、挙動の安定性と再現性を確保するという観点から、実務上は常に記述すべき要素と位置づけられる。この宣言が存在する場合、ブラウザは文書を標準モードで処理する。標準モードでは、HTMLやCSSの仕様に基づいた一貫性のあるレイアウト計算が行われるため、開発者が意図した通りの表示が再現されやすい。一方、この宣言が省略された場合や不適切な形式で記述された場合、ブラウザは互換モードに移行する可能性がある。互換モードでは、過去のブラウザとの互換性を保つために古い仕様や独自挙動が再現されるため、同じコードであってもレイアウトやスタイルの適用結果が変化することがある。

### テンプレート機能
jekyllを用いたビルドであれば、Liquid構文で_includeディレクトリ内のテンプレートを活用できる。jekyllを使わない静的なビルドの場合は、template要素によって代替できる。この代替手法はcloneNode(true)によりインライン要素間の空行を無視できる。これはtemplateがDocumentFragment内だけの処理に閉じており、ページのレイアウト計算に影響しないから。div要素で対応してもtrim()だけでインライン要素間の空行を無視できないことに起因する。templateは文書構造でしかないからscriptを書いても実行しない。インラインやブロック問わずscriptついて実行できないのは、「ページロード時に意図しないコードが走らない」というブラウザの安全設計に由来する。styleは中でも指定可能。

### Liquid構文
appendにおいてはLiquid構文は使えず、括弧を外して記述する。\
未定義変数の判定において、Liquidでは寛容なのでnullを返す。\
Liquid の変数名は 英数字とアンダースコア（_）のみ が正式にサポートされています。ハイフン（-）は演算子として「減算」として解釈されるため、変数名に入れると 構文エラー や 予期しない計算結果 を招くことがあります。代わりにアンダースコアを使うか、文字列インデックスでアクセスしてください。文字列インデックスとは、連想配列のキーを文字列で指定するとき使う形のことです。角括弧とクォートを合わせた書き方です。

## JavaScript
### クォーテーション
シングルとダブルは、外側と内側で入れ子にできる、同種ならエスケープが必要な点についても同じ。それらとは別にバッククォートは、文字列展開ができ、複数行文字列がそのまま書け、タグ付けテンプレートが使え、内部ではエスケープなしでシングルとダブルが使える。Liquid構文によるassign変数の展開についてはバッククォートでなくても動く。

### jsの読み込み順序
通常、`script` → `defer`/`module` → `DOMContentLoaded` の順でスクリプトが実行されます。`DOMContentLoaded`，`Promise`，`MutationOvserver`それぞれの挙動があるので後述します。

### DOMContentLoaded
DOM操作は div id="content" やヘッダー要素などが DOM に存在して初めて意味を持ちます。jsが body の末尾に置かれている場合、DOMContentLoadedを使ったところで、実際には DOM がすでに構築された状態で実行されますが、将来的にjs位置をヘッド側へ移したり、外部ファイル化したりする可能性 を考慮すると、DOMContentLoaded にハンドラを登録しておく方が安全です。イベントが発火すれば、ページの構造が変わっても要素取得が失敗するリスクを回避でき、コードの保守性が向上します。

### ハンドラの残留干渉
イベントハンドラは「ページがリロードされる」か「 `removeEventListener` が呼ばれる」までメモリに残ります。したがって、画面を切り替えるだけ（要素を `display:none` にしたり別の `<section>` を表示したり）では、前の画面で登録した `keydown` やクリックのリスナはそのままドキュメント全体にバインドされたままです。結果として、ユーザーが次の画面で同じキーを押すと、前画面のロジックが再び実行され、二重遷移や誤った状態更新が起こります。この干渉を防ぐには、画面を離れる直前に `removeEventListener` でリスナを外すか、イベントを特定のコンテナに委譲してそのコンテナが非表示になると自動的に届かなくなるようにし、さらに変数やフラグをリセットして前画面の状態が残らないようにします。これにより、各画面で必要なハンドラだけが有効になり、不要な処理が残って干渉することはなくなります。

### Promise
Promiseとは非同期処理の結果を格納する空箱のようなもので、未来に確定する値を表すオブジェクトであり、JavaScriptがシングルスレッド環境でも非同期処理を停止せず扱うための仕組みである。Promiseはマルチスレッドではなく、非同期状態管理を行う。［fetch，動的import，async関数］などはPromiseを返す。awaitは「箱の中身が決まるまで待ちます」と言っている。async/awaitはPromiseを同期処理っぽく扱う糖衣構文。async/await登場以前は，fetch(...).then(...)のようにPromiseチェーンで扱われていた。

### MutationOvserver
テンプレートや SPA のように要素がページ描画後に動的に追加される場合、`DOMContentLoaded` 時点ではその要素はまだ存在しません。そのため **MutationObserver** を使って DOM の変更を非同期で監視し、要素が挿入された瞬間に処理を付与する必要があります。
>基本設計では使用を避けたい。「DOMがどう変わるか分からない」前提の仕組み。順序制御で対応可能なら、そちらを優先する。htmlにおいてはawait importが役立つ。

### モジュールの必要性
js内でimportやexportを使う場合に必要である。スコープ衝突やグローバル汚染を防ぎたい場合や遅延ロードの目的（DOMContentLoadedにより異なるjs間の変数や関数の呼び出しを明示的に順序化する場合）で使う。そのため、importを含むjs内でのfetchであれば必要になる。当然ながらjsをモジュール化した場合は外部のインラインハンドラは機能しなくなるし逆も不可能。だからこそ、使いたい場合はimportやexportがある。awaitにより関数内で非同期的に停止することは可能だが、トップになると無理なのでモジュール化が必要。asyncはモジュール化不要。

### モジュール引数の必要性
html要素が設定されているならjs内部で呼び出し可能。ただし、DOM操作後に動かしたい場合や別の呼び出し機構が存在する場合には、引数を付けることで汎用性や保守性を担保できる。

### モジュール内のプライベート関数
モジュールが読み込まれればトップレベルで addEventListener が評価され、即座にリスナーが登録される。リスナーは対象オブジェクト（document、window、DOM要素）に対して動作し、export の有無は無関係。したがって、関数にまとめなくてもモジュール内部のプライベート関数を自由に呼び出せる。リスナーに対してモジュール全体にかかる変数と関数は区別して扱う。特にリスナー内部で変数を定義すると、実行されるまで存在せずロード時の実行ではエラーの原因になる。

### スコープ
ページ全体に対して1つのグローバルオブジェクトのwindowが与えられる。使用する場合は名前空間を用いる方が安全である。js自体はグローバルでhtml全体を通して定義された変数を使える。ただし、type="module"で読み込む場合は、js内の変数や関数はその内部でのみ扱うことができ、他のjsで呼び出せない。呼び出して使いたい場合はimport/exportが必要になる。この機能を使うためには同じくtype="module"である必要があるので、付け忘れに注意する。、クロージャは「内部関数 + その捕捉した環境」であり、カプセル化を可能とする。クラスはメソッド・フィールドを集合させた設計図で、# を使うと プライベートフィールド・メソッド が実装できる。関数だけでもローカル変数やクロージャで同様の隠蔽ができ、インスタンス化は不要。どちらを使っても 外部から利用する際は必ずモジュールで export し、import で取り込む 必要がある。したがって 「クラスか関数か」よりも「モジュールで公開するかしないか」 が、外部からのアクセス制御の決定要因になる。

### 要素の属性data
ある要素についてdata-csv="data.csv"のような書き方で属性を追加できる。これはデータが element.dataset に自動的にマッピングされる仕組み。

### 正規表現への対応
RegExpを使えばエスケープなしで文字列検索に対応できる。動的に生成した正規表現自体を変数に格納して使える。

### ファイル一覧の取得
事前設定ファイル数や処理を増やしたくなければHEAD方式の命名規則は連番対応とする。ブラウザはファイル列挙できないため、事前に一覧を生成しておく必要があり、その生成役がNode.js/jekyll/Actionsである。生成済みなら静的環境でも配信可能。

### 配列の削除
popで最後尾から削除したり、spliceをつかう。spliceでは削除方向を±で指定、開始位置を添え字引数で指定でき、別引数で何文字取り出すかを決める。

### ガベージコレクタ
ガベージコレクタ（GC）は，「プログラムからもう到達できなくなったオブジェクト」を自動的に回収し，メモリを再利用可能にする仕組みです。JavaScriptでは，変数やクロージャ，DOMなどから参照されているオブジェクトは生存中とみなされます。逆に，どこからも参照されなくなると「到達不能」と判定され，GCの回収対象になります。

## CSS
### IDとクラス
個別の要素に設定を反映する場合はID、複数の要素の場合はクラスをつかう。特に装飾においてクラスをつかう。

### ビューポートの扱い
bodyの高さを指定する場合は、htmlの高さに紐付けして書く方が良い。htmlの高さを指定しない場合、暗黙的にheadの高さが決まるため、bodyの位置がずれるから。

### flexの扱い
bodyすなわち親側でflexに設定すると子要素の中央寄せが簡単にできる。さらに、子要素がfixedだった場合には、子要素の位置設定が優先されるので中央寄せしたい要素と区別して扱うことができる。

### ボタン
`button`要素には，CSSを書かなくてもブラウザ側が最初からスタイルを与えています。これは「ユーザーエージェントスタイルシート（UA stylesheet）」と呼ばれるもので，単なる文字表示ではなく，「押せるUI部品」として扱うための調整が含まれています。\
そのため，ボタンのサイズは単純に「文字サイズ＋padding」だけで決まっているわけではありません。\
特に影響が大きいのは`line-height`です。多くのブラウザでは`button`に`line-height: normal;`が設定されており，この normal は固定値ではなく，フォントやOSによって自動計算されます。結果として，文字サイズを16pxにしても，実際の高さは20px前後になることがあります。

## パス指定について
### 前提
相対パスか絶対パスかの選択は，「リポジトリをまたぐかどうか」ではなく URL 空間がどう定義・制御されているかで決まる。したがって，議論の基準は常に「ディレクトリ」ではなく URL である。

### 相対パス
- ［./］：省略も可能：同じDF（ディレクトリにあるファイル名を指定する場合の書き方）
  ↑moduleのjsでimportを使用する場合はパッケージ認識となるため省略不可
  ↑import は最終的には常にURLを解決している仕組みであり、ブラウザはファイル構成やリポジトリ構造を一切知らない
- ［../］：上位階層のDF
- ［../../］：2つ上の階層のDF
  ↑相対パスの場合、呼び出し元（パスコードを記入するファイル）からの位置指定である

### ブラウザ解決規則
例えばindex.htmlからmoduleのmain.jsを呼び出しfetchを使用する場合、htmlから見た相対パスを記述する

### 相対パスの補足説明
相対パスは「今見ている URL を基準」に解決されるため，表示 URL が変わると 同じ HTML / JS でも読み込み先が変化してしまう。その結果，相対パスを使うと「どの URL で表示されたか」を常に意識する必要が生じ，思考負荷と事故の余地が増える。

### GitHub Pagesのパス構造
GitHub Pages などでは，import は GitHub リポジトリを見ているのではなく，公開された URL をそのまま取得している。URL とファイル配置が一致している静的環境では，相対パスは自然で直感的に使える。

### CDNの活用
GitHub Pages と CI/CD 分離構成を前提にすると，CDN運用の本質的な利点は，「サイトのHTML構造」と「静的アセットの配信構造」を切り離せる点にある。特に `baseurl` を用いて画像やCSSの参照問題を解決しようとすると，本来はアセット配置の問題であるにもかかわらず，Jekyll 全体のルーティング規則にまで影響が及ぶ。結果として，［CSS，JavaScript，内部リンク，OGP，canonical URL］などが相互依存し，deploy 構成変更時に破綻しやすくなる。CDN を介する設計では，HTML 側は「ページ構造とルーティング」，CDN 側は「アセット配信」という責務分離が成立するため，CI repo と CD repo が分離されていても deploy 構成に引きずられにくい。

### 絶対パス
- ［/］：リポジトリの置いてあるサイトルートDF
- ［<base>］：設定パスを基準として［/］と同じ働きをするが、特にモジュール内部でimportする場合など扱いが煩雑になりがちなので、出来るだけ使用を避ける
  ↑kramdownの脚注の参照位置がずれる
  ↑URL基準（リンクや画像の相対パス）を決めるだけで、カスタム属性の属性は触れない

### Jekyll使用時の絶対パス
- ［baseurl］：リポジトリ直下を既定値とし_config.ymlに記述してhtmlで設定可
  ↑独自ドメインを使用する場合［baseurlは空，PagesのURLは変更］になるので別の変数を設定する
  ↑フロントマターのあるファイルにのみ機能する

＊呼び出し元がhtmlでもjsでも基本は同じ書き方
＊ビルド設定によって公開ディレクトリを変更した場合、相対パス指定も同調してずれる

### 絶対パスの補足説明
ルーティングと asset 読み込みを分離するため，動的 HTML を使う環境では origin 基準（/ 始まり）のパスを使う設計が好まれる。これは「絶対パスが正しい」という意味ではなく，表示 URL に依存しない参照点を選んでいるにすぎない。フレームワークやバンドラーは，この分離を前提に asset の URL を一元管理し，開発者が表示 URL を意識しなくて済むようにしている。

### ルーティングのパス構造
ルーティングを持つシステムでは，ブラウザに表示される URL は必ずしも実在するファイルの場所を指していない。SPA や SSR では，/posts/123 や /user/me のような URL は，ファイルではなく ルーターの規則によって動的に生成された表示用パスである。この場合，HTML や JS が実際に配置されている場所（例：/assets/app.js）と，表示 URL の階層は無関係になる。

### バンドラーのパス構造
バンドラーを使う場合，開発時の import はファイルシステム（プロジェクトルート，ディレクトリ構成）を基準に解釈され，その結果がビルド時に URL ベースの参照へ変換される。# や @ などでプロジェクトルートを指す書き方は JavaScript やブラウザの仕様ではなく，バンドラーが独自に解釈しているだけである。バンドラーで「絶対パス風」に書いていても，それは URL の絶対パスではなく，ビルド前の便宜的な記法にすぎない。バンドラーを使わずにブラウザが直接 import を解釈する場合，使えるのは相対 URL，origin 基準の絶対 URL（/ 始まり），完全 URL のみである。このときの「絶対」は プロジェクトルートではなく origin 基準であり，/ は「配信元のルート」を意味する。

# マークダウン
- Marked：テスト用
- Kramdown：ビルド用
  - 要素の属性
    - インライン：最後の識別子の直後に付ける
    - ブロック：最後の識別子から改行して付ける
      - コードブロック：preのクラス設定は親側で行う

＊Jekyll同梱のkramdownがビルド前提なので用途分け必要

## 種類別の詳細
- 原始Markdownは仕様が曖昧だったため，後にCommonMarkで標準化が試みられた。その後も各実装が独自拡張を行っている。
- GFMはCommonMarkベースだが，GitHub独自拡張や実装上の調整を含む。
- MarkedやKramdownはMarkdown実装であり，実際の挙動は各実装やオプション設定に依存する。Markedでは gfm:true により GitHubライクなHTML/Markdown混在挙動が有効になり，さらに marked 実装自体が details を完全HTML block停止扱いしていないため，details内部Markdownが動いて見える。
- Kramdownの markdown="1" は，HTML block 内でも Markdown解析を継続する指定。
- details内部でMarkdownを安定して使うには，summary後や block 要素前後に空行を入れるほうが安全。details や summary の見た目変更はCSS側で調整する。summary未指定時の表示はブラウザ実装依存。
- ソフトラインブレークを `<br>` 化するには，Markedなら breaks:true，Kramdownなら hard_wrap:true のような設定が必要。breaks:true は Markdown改行をHTMLの `<br>` に変換する。一方Kramdownは HTML block を比較的厳密に扱うため，hard_wrap:true は Markdown解析対象部分にしか効かない。そのため HTML要素内部の生テキスト改行維持には，white-space: pre-wrap などCSS側対応が必要になることがある。

## ハードラインブレーク
オリジナル Markdown では，行末に半角スペース2個を置くとハードラインブレークとして扱われ，HTML 出力時には`<br>`が生成される。GitHub Flavored Markdown（GFM）や CommonMark でもこの仕様は継承されている。\
一方，単なる改行（ソフトラインブレーク）は，GitHub の標準プレビューでは表示上の改行にはならず，段落内の空白として扱われる。<br>そのため，GitHub 上で明示的に改行したい場合は，行末スペース2個，行末バックスラッシュ，または`<br>`を使用する必要がある。

このように$a+b$、 $a+b$ 違いがある。\
ブロックはこのようになる。\

# キャッシュの管理
- モジュールスクリプトのキャッシュは自動的に最適化される場合が多い
- CDNのエッジキャッシュのTTLが長いことから、cdn_baseとrevisionの合わせ技でコミットSHAを元にファイル判別する仕組みとした\
  ↑この仕組みを使うとCDNに展開しているファイルを更新してビルドした時点で、_layoutを介したファイルの全てが影響を受けて最新版に更新される\
  ↑全コミットに対する関連ファイル全ビルドを避けるため、テスト実行に位置付けた\
  ↑直接パージURLを叩けば本番更新できるが、リクエスト上限があるため頻繁な更新には向かない\
  ↑パージによる更新については、反映まで数分かかる場合あり
